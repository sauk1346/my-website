# Introducción y Fundamentos de testing de software

## Pruebas de software
Son un componente crítico en el desarrollo de aplicaciones, destinadas a garantizar calidad, fiabilidad y funcionalidad de un producto de software

## Definiciones

1. **Bug:** defecto o error en el software que causa que un programa funciona de manera incorrecta o falle al realizar la función esperada
2. **Calidad:** Cuando un producto o proceso cumple con las especificaciones y expectativas. En software, se evalúa en términos de fiabilidad, eficiencia, mantenibilidad y usabilidad
3. **Ciclo de Vida del desarrollo:** Conjunto de fases, por ejemplo: planificación, análisis, diseño, implementación, pruebas, despliegue y mantenimiento
4. **Defecto:** Anomalía en el software que puede provocar que el sistema o componente no realice la función deseada
5. **Eficiencia**: Capacidad del software para proporcionar el rendimiento adecuado en relación con la cantidad de recursos utilizados bajos condiciones específicas
6. **Error:** Equivocación cometida por una persona durante la especificación, diseño o codificación de un software
7. **Fallo:** Manifestación de un defecto en el software durante su ejecución
8. **Fiabilidad:** Capacidad del software para funcionar correctamente durante un período específico bajo condiciones determinadas
9. **IEC:** *International Electrotechnical Commision*. Organización global que prepara y publica estándares internacionales para todas las tecnologías eléctricas, electrónicas y relacionadas
10. **Inspección:** Revisión detallada de un producto de trabajo realizada por un equipo para identificar defectos y problemas de calidad
11. **ISO:** *International Organization for Standarization*. Organización internacional que desarrolla y publica estándares internacionales en una variedad de disciplinas, incluyendo la ingeniería de software y la gestión de calidad
12. **Mantenibilidad:** Facilidad con la cual el software puede ser modificado para corregir defectos, mejorar el rendimiento u otras características o adaptarse a un entorno cambiado
13. **Modularidad:** Grado en que un sistema o programa está compuesto de módulos componentes separados, pero interconectados. Permite realizar cambios en una parte del sistema sin afectar otras partes
14. **Requisitos:** Descripciones detalladas de las funciones, capacidades y atributos que debe tener un sistema o componente para satisfacer las necesidades del usuario y cumplir con las especificaciones. Incluye tanto requisitos funcionales como no funcionales
15. **Testing de software:** Ejecutar un sistema o componente con la intención de encontrar errores
16. **Usabilidad:** Facilidad con la que los usuarios pueden aprender a utilizar el software, operarlo de manera eficiente y ser satisfechos con su uso. Se centra en la interfaz de usuario y la experiencia del usuario en general
17. **Validación:** Evaluar un sistema o componente durante o al final del proceso de desarrollo para determinar si cumple con los requisitos especificados y satisface las necesidades del usuario
18. **Verificación:** Evaluar los productos intermedios de trabajo de un proyecto para asegurarse que cumplen con los requisitos específicados

## Verificación
Proceso sistemático de comprobación del software durante su desarrollo

### Fases

1. Verificación de requisitos
2. Verificación del diseño  
3. Verificación del código

## Validación
- Proceso utilizado para evaluar el software de acuerdo con las demandas del negocio o del cliente, tanto duranteo como al final del proceso de desarrollo. 
- Este proceso revisa la aplicación final para comprobar si cumple las expectativas y los requisitos de los clientes
- Los equipos de software utilizan varios métodos de validación, como las **pruebas de caja negra** (pruebas funcionales) y las **pruebas de caja blanca** (pruebas no funcionales o pruebas de diseño/arquitectura)

### Pruebas de caja blanca
Ayudan a validar la aplicacion mediante una serie de entrada de datos predefinidas. Los probadores comparan la salida de la apliacación con los valores de entrada para verificar si el software produce los resultados esperados.

Incluyen pruebas de aceptación del usuario.

### Pruebas de caja negra
Se consideran 3 variables importantes:

- Los valores de entrada
- Los valores de salida esperado
- Los valores de salida reales

Incluyen pruebas de integración, pruebas del sistema y pruebas unitarias

## Verificación vs Validación

- **Verificación:** ¿Estamos construyendo el producto de manera precisa y según las especificaciones?
- **Validación:** ¿Estamos construyendo el producto correcto que satisface las necesidades y requisitos del cliente?

|Verificación|Validación|
|---|---|
|requisitos, código y diseño|sistema, funcionalidad, rendimiento, usabilidad|
|No incluye la ejecución de código|Requiere ejecución de código para probar funcionalidad y usabilidad|
|¿se está desarrollando el producto correcto?|¿el producto desarrollado es correcto y cumple con los requisitos del cliente?|
|**práctica estática** de revisión del diseño, el código, los documentos y los programas|**Mecanismo dinámico** de prueba y validación del producto real|
|Es una comprobación, **basada en humanos**, de archivos y documentos|Es una ejecución del programa, **basada en el ordenador**|
|Ejercicio de **bajo nivel**, que precede a la validación|Ejercicio de **alto nivel** que detecta los errores pasados por alto durante la verificación|
|Objetivo: **arquitectura de software**|Objetivo: **producto real**|
|equipo de control de calidad|Equipo de pruebas|
|Métodos: revisiones, inspecciones, comprobaciones de escritorio y recorridos|Métodos: pruebas de caja negra y pruebas de caja blanca|
|**Reduce** los defectos o fallos en la fase inicial|**Detecta** fallos que se pasan por alto durante la fase de verificación|
|Ayuda a precedir si las entradas siguen a las salidas|Ayudan a precedir si los usuarios aceptarán el producto final|
